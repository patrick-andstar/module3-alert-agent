# Demo Theater Run Isolation Design

## Context

The Module 3 demo page must prove the alert false-positive analysis pipeline with visible, repeatable evidence. The current Demo Theater has several presentation issues:

- The current conclusion can show historical database rows before the user runs a scenario.
- Scenario inputs are not visible enough for acceptance review.
- Deduplication is hard to verify because the UI mainly shows the raw request/response rather than a clear merged alert.
- Long JSON fields, especially `agent_explanation`, can overflow horizontally.
- `alert_logs`, `false_positive_library`, and `whitelist_rules` only show summary rows and cannot be inspected in detail.

This design keeps the work focused on the demo and evidence layer. It does not change the database schema, recall scoring algorithm, or the main Eino Agent logic.

## Goals

- Make every main scenario generate its own isolated run.
- Ensure the current conclusion panel only reflects the latest clicked scenario run.
- Promote deduplication to a main scenario.
- Show complete request and response evidence for every scenario step.
- Represent deduplication as a real merged alert event with a new `event_id`.
- Allow row-level detail inspection for all three evidence tables.

## Non-Goals

- Do not change the MySQL table schema.
- Do not add a `run_id` column.
- Do not change structured recall weights or thresholds.
- Do not change Eino ADK or ReAct Agent behavior.
- Do not model Module 2 filtering behavior beyond simulating Module 2 event submission.

## Main Demo Scenarios

The main scenario buttons become:

1. Whitelist drop
2. Dedup window merge
3. False-positive recall and Agent judgement
4. True alert

The old "prepare demo data" action is removed from the main flow. The false-positive scenario performs its own preparation step internally:

1. `POST /api/false-positives` creates or updates the demo false-positive pattern for the current run.
2. `POST /api/client/events` submits a matching alert.
3. `POST /api/alerts/query` retrieves the alert evidence for the current run.

## Run Isolation

Each click of a main scenario generates a frontend-only `run_id`, for example:

```text
demo-20260623-153012-a7f3
```

`run_id` is not a production field and is not stored in a dedicated database column. It is encoded into generated event IDs:

```text
evt-demo-20260623-153012-a7f3-01
evt-demo-20260623-153012-a7f3-02
merge-demo-20260623-153012-a7f3-001
```

Production semantics remain:

- Raw `event_id` values are client/module-2 event IDs.
- Merged alert `event_id` values are generated by Module 3 because deduplication creates a new aggregate alert.
- `run_id` is only a Demo Theater correlation mechanism.

The current conclusion panel must never fall back to `alerts[0]` or other historical rows. Before any scenario runs, it shows an empty waiting state. After a scenario runs, it displays only records whose `event_id` belongs to the current `run_id`.

The lower historical tables may still show all recent database records. They are evidence browsing views, not the source of the current scenario conclusion.

## Dedup Merge Event

The dedup scenario submits 10 raw events with the same merge key:

```text
host_id + user_id + process_name + sensitive_type + operation
```

The expected final evidence is one merged alert event:

```text
merge-<run_id>-001
```

The merged alert must be a complete `model.Event`, not just an original event with a visual label. It keeps shared fields such as:

- `host_id`
- `user_id`
- `process_name`
- `process_path`
- `sensitive_type`
- `operation`
- `target`
- `risk_level`

It also includes:

- `is_merge_event: true`
- `file_count: 10`
- `files`: 10 `{file_path, file_hash}` entries
- `time_range`
- `duration`

The existing `alert_logs.files_json` field stores the `files` array. No schema change is required.

The UI should emphasize the merged alert. The raw 10 events remain visible in the request evidence for traceability, but they should not be presented as 10 final alerts.

## Step Evidence UI

Each scenario step is displayed as an expandable block.

Collapsed state shows:

- Step number
- Label
- HTTP method
- Path
- Status code
- Key outcome summary

Expanded state shows request and response evidence using tabs or a two-column layout:

- `Request`: method, path, headers if relevant, full body
- `Response`: status, full response body

JSON output must wrap long lines. Long fields such as `agent_explanation` must remain readable without horizontal scrolling.

For the dedup scenario, the evidence area additionally highlights:

- merged `event_id`
- `file_count`
- `files[]`
- `time_range`
- `duration`

## Table Detail Views

The lower evidence tables remain:

- `alert_logs`
- `false_positive_library`
- `whitelist_rules`

Each row becomes clickable and opens a detail panel or modal. The detail view has two layers:

1. Business detail
2. Raw JSON

`alert_logs` business detail shows:

- `event_id`
- risk change
- `agent_verdict`
- `agent_confidence`
- `recall_score`
- `agent_explanation`
- `false_positive_reason`
- `is_merge_event`
- `file_count`
- `files[]`

`false_positive_library` business detail shows:

- `scenario_key`
- `sensitive_type`
- `operation`
- `process_name`
- `process_path`
- `target`
- `reason`
- `hit_count`
- `last_seen_at`
- `expired_at`

`whitelist_rules` business detail shows:

- `rule_name`
- `logic`
- enabled state
- configured match fields
- time window fields

Raw JSON displays the full row object in all three detail views.

## Data Flow

1. User clicks a main scenario.
2. Frontend generates `run_id`.
3. Frontend creates scenario steps with event IDs containing `run_id`.
4. Frontend clears current conclusion, then executes the steps in order.
5. After event submission, frontend queries alerts and filters records by the current `run_id` encoded in `event_id`.
6. Current conclusion panel renders only the filtered current-run record.
7. Historical tables refresh independently and may show all recent rows.

## Error Handling

- If any step returns HTTP 400 or above, mark the run as failed and keep the failed request/response visible.
- Do not replace the current conclusion with historical data after failure.
- If the filtered query returns no current-run records, show a "no result for this run" state with the executed requests still visible.
- The whitelist drop scenario may have no alert row by design; it should show accepted/dropped evidence from the event response instead of searching for an alert row.

## Testing And Verification

Frontend checks:

```powershell
cd module3-alert-agent/frontend
npm run typecheck
npm run build
```

Backend checks:

```powershell
cd module3-alert-agent
go test ./...
```

Manual acceptance:

- First page load shows no current conclusion from historical data.
- Each main scenario click creates a fresh `run_id`.
- All main scenarios show request and response evidence.
- Dedup scenario submits 10 raw events and displays one merged alert with `merge-<run_id>-001`, `file_count: 10`, and `files[]`.
- Long JSON lines wrap inside the evidence output.
- Each of the three evidence tables supports row-level business detail and raw JSON inspection.
- Recall scoring behavior remains unchanged.
- No database migration or new table column is introduced.
